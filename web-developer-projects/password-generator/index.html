Password generator redesign

A redesigned password generator based off an old bootcamp assignment.

Project Overview

Summary

Having worked as a Web developer for a few years now, I wanted to challenge myself and see if I could redesign an old bootcamp project - a password generator.

The goal

My goal for this project was to go through the original project, identify opportunities for improvement, and redesign the application and its functionality to provide a better user experience.

First steps

Identifying opportunites for improvement

One of the first things I did, was go through the password generator and identify some opportunities for improvement. To see what the original application looked like, click here. In going through the application, I found a few things that could be improved:

Blank password

With the exception of the prompt asking for password length, if a user doesn't select any password options (they select "Cancel" in all the prompts), a blank password is provided to the user.

An inability to go back if a user changes their mind

The questions asking what a user wants in their password are asked sequentially. If a user changes their mind, they're unable to go back to the option they passed on. The only way they could go back, is to finish creating a password and then create another one, or refresh the page and start over.

The overall design

The design of the application was already made, as the assignment mainly focused on creating the functionality. The application also uses JavaScript prompts to ask what the user wants in their password.

Lack of a copy feature

The generator lacks a copy feature so a user would have to manually copy their password.

Tackling the problems

The overall design

In coming up with the new design, one of the things I wanted to address was the default JavaScript prompts. Thinking about the user and how the prompts didn't allow a user to go backwards, I opted to use checkboxes so a user could select and unselect their options. The checkboxes informed how I was going to design the rest of the application. With the checkboxes in mind, I created a few wireframes and added stars to the particular sections that I liked.

Addressing how a user selects password options

With the checkboxes now part of the user interface, a user can freely select their password options without the constraints provided by the JavaScript prompts.

Preventing a blank password from populating

To address the blank password problem, I added some "if" statements to my JavaScript code that checks to see if at least one option was selected. If none of the options are selected, a warning message appears to the user and a password isn't made. To see how I solved this, click here.

Adding a copy feature

In order to add the copy feature, I used the Clipboard API. I added an event listener on the password box, and when clicked, would copy the password box contents to the clipboard.

Challenges

When redesigning this application, I faced a few challenges: determining which options were selected after switching to checkboxes and creating a password that wasn't the length the user asked for.

Challenge #1 - Determining which of the checkboxes were selected

Now that the generator uses checkboxes, I had to see which options were selected. My solution involved the following steps:

  1. Create an object to hold the password options, and create a variable to hold the number of options that weren't selected (a counter).
  2. Check to see which boxes were selected using the ".checked" property.
  3. Loop through all of the checkboxes and get the status of each, and adding the information to the "password options" object.
  4. After adding the information to the password options, loop through the object using a "for...in" loop to see which options are "false".
  5. If an option is "false", add 1 to the variable that keeps track of the "false" options.
  6. If the counter equals 4, no options were selected so warn the user that they need to select at least one option.

Challenge #2 - Incorrect password length

After switching to checkboxes and creating a password, the final password length was actually multiplied by however many options were selected. In the GIF provided, the password length is noted in the right dark gray area at the end of the password (the number). In this example, if the password length is 128 characters in length and 4 options are selected, the final password is 512 characters in length; if the password length is 128 and 2 options are selected, the final password is 256 characters in length.

The problem was in the JavaScript code that checks the password length (the “if” statement). The condition the code was checking for was very strict in that as the password was being generated, if the user’s password length didn’t exactly match the length of the generated password during one of the length checks, then the code would continue to add characters to the password.

To fix this, the condition was revised so that if the generated password was greater than the user's requested password length, the generator stops adding characters.

Learnings and future

Learnings

Working on this project, I learned how to loop through a JavaScript object using a “for...in” loop. I also learned how to redesign an app and its functionality from pre-existing code in order to provide a better user experience. Finally, I learned how to use the Clipboard API in order to create basic copy functionality.

Future

If I were to continue improving this project, I'd do these things:

  • I would conduct a usability study to see what users think
  • Add a feature that would allow users to save their passwords
  • Use Figma to create mockups that I can reference during development
